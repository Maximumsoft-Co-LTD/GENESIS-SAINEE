## 5-Prompt Framework (Top-Down ทำมือ)

เทคนิคขุด legacy code เป็น 4 เลเยอร์ (Top-Down) ไล่จากโครงสร้างภายนอก → ท่อส่งข้อมูล
→ ตรรกะ → คัดแยกขยะ ชุด prompt สำเร็จรูปที่ก๊อปไปใช้ได้เลย

> 💬 **ความเห็น (โดยรวม):** เฟรมเวิร์กนี้ดีและใช้ได้จริง โดยเฉพาะลำดับ Top-Down ที่ลด
> "ความไม่รู้" ได้เร็วที่สุด (รู้ entry point ก่อน แล้วค่อยเจาะลึก) และ Layer 5
> (Keep/Drop) ที่คนมักลืม
>
> **แต่:** เฟรมเวิร์กนี้ออกแบบมาสำหรับ "ป้อน code เอง" ใน chat ที่ไม่มี filesystem
> access ส่วนใน Claude Code เรามี `/archaeology` skill ที่ทำ multi-agent อ่านทั้ง
> repo อัตโนมัติอยู่แล้ว ดังนั้นควรใช้ 5 prompts นี้เป็น **"checklist ว่า output
> ต้องครอบคลุมอะไร"** มากกว่าทำมือทีละขั้น

---

### 📂 Layer 0 (เพิ่มเอง): Runtime / Ops Reality

> 💬 **ความเห็น:** ก่อน Layer 1 ควรเข้าใจ "ระบบรันจริงยังไง" เพราะ "โค้ดทำอะไร" กับ
> "ระบบจริงทำงานยังไง" มักไม่ตรงกัน
>
> **ป้อนให้ AI:** env vars (`.env`), `Dockerfile`, k8s manifest, config ที่ต่างกัน
> ระหว่าง dev/uat/prod
>
> **ถาม:** ระบบนี้ deploy ยังไง? มี config ไหนที่เปลี่ยนพฤติกรรมตาม environment?
> มี secret/connection string ชี้ไปที่ไหนบ้าง?

---

### 📂 Layer 1: สแกนโครงสร้างและระบุ "หัวใจ" (Entry Points)

**ป้อนให้ AI:** โครงสร้างโฟลเดอร์ (Folder Tree) + ไฟล์ตั้งค่าหลัก เช่น
`package.json`, `go.mod`, `pom.xml`, `Dockerfile` หรือ `main.js/go/py`

```
ฉันกำลังทำ System Archaeology เพื่อย้ายระบบเก่าไปรักษาระบบใหม่ นี่คือไฟล์
Configuration และโครงสร้างโฟลเดอร์ของ Repo นี้ ช่วยวิเคราะห์และสรุปให้หน่อย:

- Repo นี้ใช้ Tech Stack อะไร (เวอร์ชันไหน) และทำหน้าที่หลักอะไรในระบบ?
- จากโครงสร้างไฟล์ จุดไหนคือ Entry Point (จุดเริ่มต้นการทำงาน) ของระบบนี้?
- มี Library หรือ Dependency ตัวไหนที่สำคัญ หรือดูแปลกปลอม/เก่าเกินไปที่ฉันควรระวังบ้าง?
```

---

### 🔄 Layer 2: แกะรอยทางเข้าออก (API Contracts & Data Flow)

**ป้อนให้ AI:** ไฟล์จำพวก Router, Controller หรือไฟล์ที่ประกาศ Endpoint ของ API

```
นี่คือไฟล์ที่จัดการเรื่อง Route/Controller ของ Repo นี้ ช่วยลิสต์ Endpoint ทั้งหมด
ออกมาในรูปแบบตาราง โดยระบุ:

- HTTP Method และ Path (เช่น POST /v1/orders)
- หน้าที่ของ Endpoint นั้นในเชิงธุรกิจ
- มีการรับค่า (Input/Payload) และส่งค่ากลับ (Output) หน้าตาประมาณไหน?
- มี Endpoint ไหนไหมที่ดูเหมือนไม่ได้ใช้งานแล้ว (Dead Endpoint) หรือเขียนทิ้งไว้เฉยๆ?
```

> 💬 **ความเห็น:** เคสทีมเรามี RabbitMQ/Redis เยอะ ลำพัง HTTP endpoint ไม่พอ ต้อง
> map **async paths** ด้วย — queue consumer/producer, callback, scheduled job
> ลองเพิ่มคำถาม: "นอกจาก HTTP มี message queue / event / cron ตัวไหนที่เป็น entry
> point หรือ trigger งานบ้าง? แต่ละตัวรับ payload อะไร แล้วส่งต่อไปไหน?"

---

### 🧠 Layer 3: ชำแหละสมองและกับดัก (Business Logic & Edge Cases)

**ป้อนให้ AI:** ไฟล์ Service, Domain Logic, หรือโค้ดที่มีเงื่อนไข if-else เยอะ
(ทีละไฟล์/ทีละฟังก์ชันสำคัญ)

```
ฉันต้องการถอดสมอง (Business Logic) จากโค้ดไฟล์นี้ เพื่อเอาไปเขียนใหม่ในภาษาอื่น
ช่วยอธิบายตรรกะการทำงานของฟังก์ชัน [ระบุชื่อฟังก์ชัน] ออกมาเป็น Pseudocode ภาษาไทย
ที่เข้าใจง่าย โดยเน้นย้ำจุดเหล่านี้เป็นพิเศษ:

- มี Edge Case หรือเงื่อนไขแปลกๆ (เช่น การเช็กค่า null, การดักจับ Exception)
  อะไรบ้างที่ซ่อนอยู่?
- มีโค้ดส่วนไหนที่เป็น Workaround (เขียนแก้ขัด) หรือเป็นหนี้ทางเทคนิค
  (Technical Debt) ที่ฉันไม่ควรลอกไประบบใหม่ไหม?
```

> 💬 **ความเห็น:** Pseudocode ภาษาไทยดีสำหรับให้คนอ่าน แต่ถ้าจะ migrate จริง
> ความกำกวมของภาษาธรรมชาติทำให้ edge case หล่นได้ แนะนำให้ AI ออก **decision table**
> หรือ **given/when/then** ควบคู่ไปด้วย เพื่อให้เงื่อนไขทุกกิ่งถูกระบุชัดเจน

---

### 🗄️ Layer 4: เจาะลึกการเก็บข้อมูลและเชื่อมต่อภายนอก (Data & Integration)

**ป้อนให้ AI:** ไฟล์ Schema, Models, Migration หรือโค้ดที่เรียก Database, Redis,
Kafka และ Third-party API

```
นี่คือส่วนที่ระบบคุยกับฐานข้อมูลและบริการภายนอก ช่วยสรุปให้หน่อยว่า:

- โครงสร้างข้อมูล (Data Model) ของไฟล์นี้มีความสัมพันธ์กับโต๊ะ/คอลเลกชันอื่นอย่างไร?
- มีการใช้ Cache (เช่น Redis) ในจังหวะไหนบ้าง และมีนโยบายการเคลียร์ Cache อย่างไร?
- ระบบนี้เชื่อมต่อไปยัง Third-party หรือ Service อื่นข้างนอกอย่างไร
  (ใช้ Auth แบบไหน, ส่งข้อมูลอะไรไป)?
```

---

### 🧹 Layer 5: สรุปใบคัดแยก (Keep / Drop Summary)

หลังคุยครบทุกเลเยอร์ ตบท้ายด้วย prompt นี้เพื่อทำรายงานส่งต่อ:

```
จากข้อมูลทั้งหมดที่เราคุยกันเกี่ยวกับ Repo นี้ ช่วยสรุปเป็นรายงานสั้นๆ ให้หน่อยว่า:

- ฟีเจอร์หรือตรรกะอะไรบ้างที่ 'ต้องลอก' ไประบบใหม่แบบห้ามตกหล่น (Core Business Value)?
- ส่วนไหนบ้างที่ 'ควรโละทิ้ง' หรือเขียนใหม่ทั้งหมด
  (เช่น โค้ดเก่าเกินไป, วิธีการเขียนล้าสมัย)?
```

> 💬 **ความเห็น:** Layer นี้สำคัญที่สุดและคนมักข้าม — มันเปลี่ยนจาก "อ่านโค้ดเฉยๆ"
> เป็น **"ตัดสินใจเชิงสถาปัตยกรรม"** ว่าจะเก็บหรือทิ้งอะไร ควรทำทุกครั้งก่อนเริ่ม
> migrate จริง
